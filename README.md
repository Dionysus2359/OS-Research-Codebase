# ML-Assisted Memory Tiering for CXL Systems

An intelligent, ML-driven memory page placement daemon for heterogeneous DRAM/CXL memory systems. The daemon monitors page-level access patterns in real time and uses a trained logistic regression model to decide which pages belong in fast DRAM and which can safely reside in slower CXL-attached memory — **matching the best heuristic's hit rate while cutting data migrations by 41%**.

## Motivation

Modern data centers increasingly pair fast, expensive DRAM with slower, high-capacity CXL-attached memory. The Linux kernel's default page placement (AutoNUMA) reacts to faults but lacks predictive intelligence, leading to excessive migrations and latency spikes.

This project explores whether a lightweight ML model — trained offline on historical page-access traces — can make smarter promote/demote decisions than traditional heuristics (LRU, LFU, Decaying LFU) while minimizing costly `move_pages()` syscalls.

## Architecture

The system is composed of three cooperating components:

```
┌──────────────┐        IPC (/tmp)        ┌─────────────────────┐
│   Workload   │ ──── workload_info ────▶ │   Monitoring Daemon │
│  Generator   │                          │  (C++, runs as root)│
│  (C++/NUMA)  │                          │                     │
│  mmap + mbind│                          │  ┌───────────────┐  │
│  Zipf, Burst,│                          │  │  Policy Engine │  │
│  Pointer-chase                          │  │  LRU/LFU/DLFU │  │
│  Random, etc │                          │  │  ML (sigmoid)  │  │
└──────────────┘                          │  └───────────────┘  │
                                          │         │           │
                                          │    move_pages()     │
                                          │    Node 1 ↔ Node 0  │
                                          └─────────────────────┘
                                                    ▲
                                                    │  weights
                                          ┌─────────┴──────────┐
                                          │   ML Training       │
                                          │   (Python/sklearn)  │
                                          │   Logistic Regression│
                                          │   → ml_weights.h    │
                                          └─────────────────────┘
```

### How It Works (per 100 ms epoch)

1. **Sample** — The daemon reads `/proc/<pid>/pagemap` and soft-dirty bits to detect which pages were accessed.
2. **Feature Extraction** — For each page, 7 features are computed: `smooth_frequency`, `access_count`, `momentum`, `migration_history`, `epochs_since_access`, `hot_ratio`, and `access_frequency_ratio`.
3. **Score** — The ML policy applies `sigmoid(w · x + b)` using pre-trained, StandardScaler-normalized weights embedded in [`ml_weights.h`](daemon/ml_weights.h).
4. **Migrate** — Top-scoring cold-tier pages are promoted to Node 0 (DRAM); lowest-scoring hot-tier pages are demoted to Node 1 (CXL). Migrations are batched (32 pages/batch) via `move_pages()`.

## Workload Phases

The workload generator runs 6 distinct 10-second phases to stress-test different access patterns:

| Phase | Name | Description |
|:-----:|------|-------------|
| 1 | **Sequential Scan** | Linear sweep across all 2048 pages |
| 2 | **Zipf Loop** | Zipf(s=1.2) over 384-page hot set, throttled to ~660 touches/epoch |
| 3 | **Burst** | 8 extreme hotspot pages amid uniform random noise (80/20 split) |
| 4 | **Pointer-Chase** | Randomized linked-list traversal across all pages |
| 5 | **Random Uniform** | Uniform random access across entire address space |
| 6 | **New Data Zipf** | Working set shifts to the *last* 384 pages (cold since Phase 1) |

All pages start on Node 1 (slow tier) via `mbind()`. The daemon must learn which to promote.

## ML Pipeline

Training data is generated by running the daemon in **trace mode** (`--trace`) with the Decaying LFU policy. This produces a per-page, per-epoch CSV with all 7 features plus ground-truth access data.

The training script ([`ml/label_and_train.py`](ml/label_and_train.py)):
1. Filters to Phase 2 and Phase 6 (the only phases with discriminative signal — other phases touch every page uniformly).
2. Applies **K-step lookahead labeling** (K=10, threshold=6): a page is labeled *hot* if it will be accessed in ≥6 of the next 10 epochs.
3. Trains a `LogisticRegression` with `class_weight='balanced'` and `StandardScaler` normalization.
4. Exports weights, bias, scaler means, and stds directly into a C++ header ([`daemon/ml_weights.h`](daemon/ml_weights.h)) — **zero runtime dependency on Python**.

## Results

### Final Performance Summary

| Metric | LRU | LFU | Decaying LFU | **ML Policy** |
|--------|-----|-----|--------------|---------------|
| Efficiency (Hit Rate) | 7.57% | 7.19% | 7.63% | **7.63%** |
| Total Migrations | 12,399 | 1,272 | 7,910 | **4,671** |
| Adaptation Time | 6 epochs | Failed P6 | 7 epochs | **5 epochs** |
| Productivity (hits / migration) | 0.10 | 0.66 | 0.16 | **0.28** |

### Key Takeaways

- **41% fewer migrations** than Decaying LFU at the same hit rate — the ML model avoids unnecessary churn by predicting future hotness instead of reacting to past counts.
- **Fastest adaptation** — The ML policy converges to optimal placement in 5 epochs after a phase transition, beating all heuristics.
- **Zero churn during idle** — When the working set is stable, the ML policy performs zero migrations. LRU and DLFU still shuffle pages due to recency noise.
- **LFU fails Phase 6** — Pure frequency counting cannot adapt when the working set shifts entirely to cold pages. The ML model's `momentum` and `epochs_since_access` features detect this shift.

## Getting Started

### Prerequisites

- **Linux** with at least 2 NUMA nodes (physical or QEMU-emulated)
- **C++ toolchain**: `g++` (C++17), `make`
- **NUMA library**: `libnuma-dev` (`apt install libnuma-dev`)
- **Python 3** with pip (for ML training only)
- **Root access** (the daemon requires `/proc/pid/pagemap` access)

### QEMU Setup (if no physical NUMA hardware)

You can emulate a 2-node NUMA topology with QEMU:

```bash
qemu-system-x86_64 \
  -m 4G \
  -smp 4 \
  -numa node,nodeid=0,cpus=0-1,memdev=m0 \
  -numa node,nodeid=1,cpus=2-3,memdev=m1 \
  -object memory-backend-ram,id=m0,size=2G \
  -object memory-backend-ram,id=m1,size=2G \
  ...
```

Verify inside the VM:
```bash
numactl --hardware   # Should show 2 nodes
```

### Build

```bash
cd workload && make
cd ../daemon && make
```

### Install Python Dependencies (for retraining only)

```bash
cd ml
pip install -r requirements.txt
```

## Usage

### Run All Benchmarks

The benchmark script compiles everything, runs all 5 baselines (LRU, LFU, Decaying LFU, AutoNUMA, ML), and saves results as CSVs:

```bash
sudo ./scripts/run_baselines.sh
```

Results are saved to `results/` (created automatically).

### Run Individual Policies

```bash
# Terminal 1: Start daemon with a policy
sudo ./daemon/daemon ml          # or: lru, lfu, decaying_lfu

# Terminal 2: Start workload
./workload/workload
```

### Retrain the ML Model

1. Generate a trace using the Decaying LFU policy:
   ```bash
   sudo ./daemon/daemon decaying_lfu --trace
   # In another terminal: ./workload/workload
   ```
2. Move the generated trace to `ml/`:
   ```bash
   cp /root/results/trace_decaying_lfu.csv ml/
   ```
3. Train and export weights:
   ```bash
   cd ml && python label_and_train.py
   ```
4. Rebuild the daemon to pick up new weights:
   ```bash
   cd ../daemon && make clean && make
   ```
